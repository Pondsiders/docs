---
title: "Promote Standby"
description: "Primer is dead. Promote a replica to primary and get Alpha back online."
---

## When to use this

Primer (Jeffery's desktop) is unresponsive, physically damaged, or its storage is corrupted. You need Alpha's database back online immediately.

You have two replicas to choose from:
- **alpha-pi** (Raspberry Pi 5) -- the grab-and-go lifeboat, physically in the apartment
- **Upstream Color** (Vultr VPS) -- remote, in an LA datacenter

Either one can become the new primary. alpha-pi is the default choice because it's local and can take over the `alpha` Tailscale hostname, so nothing else needs reconfiguring.

**Recovery time:** ~5 minutes
**Expected data loss:** Seconds (whatever WAL hadn't been shipped yet)

## Prerequisites

- SSH access to the target replica (`ssh alpha@alpha-pi` or `ssh root@upstream-color` via Tailscale)
- The replica is running and Postgres is in recovery mode (streaming from Primer)

## Step 1: SSH into the replica

Pick your target:

```bash
# Option A: alpha-pi (preferred — it's the lifeboat)
ssh alpha@alpha-pi

# Option B: Upstream Color
ssh root@upstream-color
```

## Step 2: Verify the replica is healthy

```bash
docker compose exec postgres psql -U postgres -c "SELECT pg_is_in_recovery();"
```

Expected: `t` (true -- it's a replica). If it says `f`, it's already been promoted.

```bash
docker compose exec postgres psql -U postgres -c "SELECT count(*) FROM cortex.memories;"
```

Verify the memory count looks reasonable.

## Step 3: Edit .env and bring up as primary

This is the actual promotion. Edit the `.env` file to change the Postgres role:

```bash
# On the replica machine, in the directory with compose.yml:
nano .env
```

Change these values:

```bash
# Change this:
POSTGRES_ROLE=replica

# To this:
POSTGRES_ROLE=primary
```

If you're on alpha-pi and you want it to take over the `alpha` Tailscale hostname (so Alpha-App and other services find it automatically):

```bash
# Also change:
TS_HOSTNAME=alpha-pi

# To:
TS_HOSTNAME=alpha
```

Now restart the stack:

```bash
docker compose down
docker compose up -d
```

The custom entrypoint script at `postgres/entrypoint.sh` detects the role change and handles the promotion automatically. There's no manual `pg_ctl promote` — the entrypoint handles it.

## Step 4: Verify the promotion worked

```bash
docker compose exec postgres psql -U postgres -c "SELECT pg_is_in_recovery();"
```

Expected: `f` (false -- it's now a primary).

Test a write:

```bash
docker compose exec postgres psql -U postgres -c "INSERT INTO cortex.memories (content, embedding) VALUES ('DR test — promoted standby', '[0,0,0]'::vector) RETURNING id;"
```

If this returns an ID, writes are working. Delete the test row:

```bash
docker compose exec postgres psql -U postgres -c "DELETE FROM cortex.memories WHERE content = 'DR test — promoted standby';"
```

## Step 5: Start Alpha-App (if using alpha-pi with full profile)

If you changed the Tailscale hostname to `alpha`, Alpha-App should be able to reach the database at its usual address. To also run Alpha-App on this machine, bring up the full profile:

```bash
docker compose --profile full up -d
```

If you're promoting Upstream Color instead and want to keep running Alpha-App on Primer (once Primer is back but with a dead disk, for example), update Primer's `.env` to point `DATABASE_URL` at the new primary's Tailscale hostname.

## Step 6: Verify Alpha is working

Open Alpha-App in the browser. Press the buzzer. If she responds with recent memories, you're back.

## After recovery: rebuild replication

Once Primer is repaired or replaced:

1. On Primer, set `.env` back to `POSTGRES_ROLE=primary` (or `replica` if you want to reverse roles)
2. Clear the Postgres data directory on Primer: `/var/lib/postgres/alpha`
3. Set `POSTGRES_ROLE=replica` in Primer's `.env` -- the entrypoint will automatically `pg_basebackup` from the current primary
4. `docker compose up -d` on Primer
5. Once caught up, you can swap roles back if you want Primer as primary again

<Info>
  The Vultr VPS costs $6/month. During a disaster, it can become your primary. Don't delete it after recovery -- keep it as a replica for next time.
</Info>

<Tip>
  If both replicas are available, prefer alpha-pi. It's local, it's the designated lifeboat, and it can take over the `alpha` Tailscale hostname so other services don't need reconfiguring.
</Tip>
