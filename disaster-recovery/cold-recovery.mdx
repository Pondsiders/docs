---
title: "Cold Recovery"
description: "Everything is dead. Rebuild Alpha's database from B2 backups on new hardware."
---

## When to use this

All three machines are gone — Primer, Upstream Color, and alpha-pi. You're rebuilding from scratch using backups stored in Backblaze B2.

**Recovery time:** ~30 minutes
**Expected data loss:** Seconds to minutes, depending on WAL archiving lag

<Warning>
  Try the [Promote Standby](/disaster-recovery/promote-standby) playbook first. Cold recovery is slower. Only use this if all replicas are also dead.
</Warning>

## What you need

- A machine with Docker and Docker Compose installed (new VPS, repaired Primer, a laptop — anything)
- The Alpha-App repo (clone from GitHub: `Pondsiders/Alpha-App`, or restore from Restic)
- B2 credentials from 1Password ("Pondside" vault)

## Step 1: Download the base backup and WAL

```bash
pip install b2

# Authorize (credentials from 1Password)
b2 authorize-account YOUR_KEY_ID YOUR_APP_KEY

# List available base backups — pick the most recent
b2 ls --long alpha-yellowstone basebackup/

# Download it
b2 download-file-by-name alpha-yellowstone basebackup/FILENAME ./base-backup.tar.gz

# Download all WAL segments
mkdir -p /tmp/wal-archive
b2 sync b2://alpha-yellowstone/wal/ /tmp/wal-archive/
```

## Step 2: Restore the base backup

```bash
sudo mkdir -p /var/lib/postgres/alpha
sudo tar xzf base-backup.tar.gz -C /var/lib/postgres/alpha/
sudo chown -R 999:999 /var/lib/postgres/alpha/
```

## Step 3: Configure WAL recovery

```bash
touch /var/lib/postgres/alpha/recovery.signal

echo "restore_command = 'cp /tmp/wal-archive/%f %p'" \
  >> /var/lib/postgres/alpha/postgresql.auto.conf

# Optional: recover to a specific point in time instead of "latest"
# echo "recovery_target_time = '2026-03-29 14:30:00 UTC'" \
#   >> /var/lib/postgres/alpha/postgresql.auto.conf
```

## Step 4: Start Postgres

Set up `.env` with `POSTGRES_ROLE=primary` and start the stack:

```bash
docker compose up -d postgres
```

Postgres replays the WAL segments and recovers to the latest available point (or to the target time if you specified one). Check the logs:

```bash
docker logs alpha-postgres --tail 20
```

Look for `consistent recovery state reached` and `database system is ready to accept connections`.

## Step 5: Verify

```bash
docker compose exec postgres psql -U postgres \
  -c "SELECT count(*) FROM cortex.memories;"

docker compose exec postgres psql -U postgres \
  -c "SELECT id, left(content, 60) FROM cortex.memories ORDER BY id DESC LIMIT 5;"
```

## Step 6: Bring up the full stack

```bash
docker compose --profile full up -d
```

Press the buzzer. She should be back.

## Step 7: Re-establish replication

Once stable, set up the other machines as replicas:

1. Set `.env` to `POSTGRES_ROLE=replica` with the new primary's Tailscale hostname
2. Make sure `/var/lib/postgres/alpha` is empty
3. `docker compose up -d` — the entrypoint runs `pg_basebackup` automatically

## What you'll lose

Only data between the last archived WAL segment and the failure — typically seconds to minutes.

## What you won't lose

- All memories up to the last WAL segment
- Pondside directory tree (backed up via Restic to B2)
- All code (in git on GitHub)

<Tip>
  The `alpha-yellowstone` bucket has 7-day file versioning. Even if a backup file is overwritten or corrupted, B2 keeps previous versions for 7 days.
</Tip>
